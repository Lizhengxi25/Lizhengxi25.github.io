Writing a solution for myself—let's call it a note. Since it's written for myself, I should be honest and kind to myself. I won't make too many excuses, but I also want to leave a mark of my youth.

Luogu P1339 [USACO09OCT] Heat Wave G

When I was first learning graph theory, I read many recommendations on Zhihu and Codeforces about practicing problem-solving. Many experts suggested that when practicing, one should minimize or avoid reading solutions. If you can't solve a problem, come back to it later. Coach D also told us: Our tolerance for tough problems is much lower than it was in his time. Nowadays, if we can't solve a problem within four hours, we just give up. Back then, he would often ponder a problem for a week. Essentially, you have to think independently and develop your own methods rather than purely applying templates.

Moreover, Coach D hasn't taught our class graph theory yet. My approaches to graph problems are based on the book "An In-Depth and Simple Approach," solutions to various standard problems, and my own explorations. I don't even know the names of many algorithms. When learning about graph storage and traversal, I learned the "Forward Star" method. The rest is based on depth-first search (DFS) and breadth-first search (BFS), adapting as needed.

When I first attempted this problem, I thought it seemed standard—you just traverse all the nodes. Later, I realized the difference in this problem is that the weight of each edge is not necessarily 1. Previously, BFS or DFS had the nice property that each layer traversed increases the distance by 1. Therefore, as the traversal goes deeper, the distance increases—the distance has a monotonic relationship with the depth of traversal. Thus, the first time we reach the target node, the distance is the answer.

However, in this problem, the edge weights are not all 1, so the distance between two nodes does not have a monotonic relationship with the depth of traversal. So I thought of enumerating the distances from the starting point to each node, maintaining them using a priority queue. For example, first enqueue the adjacent nodes of the starting point, enumerate the distances, and if there's a node whose distance to the endpoint is exactly ( i ), then the shortest distance from the starting point to that node is ( i ). We can mark it as visited and enqueue it.

Before discussing optimization, let me emphasize that along a path without repeated nodes, the distance between two nodes still has a monotonic relationship with the depth of traversal because the edge weights are non-negative.

When we enumerate distances, there can be many empty distances, wasting time. However, with a priority queue, the node at the front is always the closest, so each time we only need to dequeue the front element and expand from it. When we reach the target node, the current distance is the shortest distance. Note: Remember to remove nodes that have been dequeued.
