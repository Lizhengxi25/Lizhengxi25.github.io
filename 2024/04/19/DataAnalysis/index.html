Abstract

I. Data Processing

Mathematical Part

Significant Figures and Scientific Notation

Concept

The significant figures of a datum are the total number of digits when it is written in scientific notation, used to describe the precision of the data. Distinguishing between digits that indicate the order of magnitude and significant digits is vital, and it follows these rules:

  • Rules for zeros in data: If omitting the zero changes the value of the number, then it is a placeholder zero indicating the order of magnitude and is not a significant digit (e.g., the significant figures in are 1 digit—). If omitting the zero does not change the value, then it indicates precision and is a significant digit (e.g., has 2 significant figures—).
  • When expressing numbers in scientific notation, the coefficient must satisfy and must not be a fraction.
Operation Rules
  • Addition and Subtraction: The number of decimal places in the result should match the least number of decimal places in the operands.
  • Multiplication, Division, and Exponential Operations: The number of significant figures in the result should match the operand with the fewest significant figures.
  • Rounding: The result should be rounded according to standard rounding rules.

II. Code Construction

Programs designed for data analysis should first ensure the repeatability of modules, meaning the reusability of code components. For example, an independent experiment might be conducted multiple times, and within one experiment, the independent variable might be altered several times. Therefore, how arrays are initialized, how parameters are passed, and how outputs are generated become crucial.

Data and Variables

Overview of Data Storage

Efficient and clear data transmission is the foundation of an effective program, especially in data analysis where data and variable storage are paramount. Programs are composed of countless data points, so establishing a robust data handling strategy is essential.

The primary concept is to view the program as an auxiliary tool for data analysis, not the core component of the entire experiment. A reasonable and efficient data flow involves the program reading data, performing analysis, and then outputting results. It's crucial not to mix raw data, analysis results, and program variables; these three should always remain independent. Therefore, it's advisable to store raw data and analysis results in two separate files.

Avoid having the program write directly to the raw data file. Programs can have uncertainties, and altering raw data contradicts the principle of maintaining its integrity. Even well-tested programs should not modify raw data directly, as evolving requirements and ideas necessitate preserving the purity of the original dataset for future iterations.

Also, do not embed raw data within the program code. Firstly, most programming languages are less intuitive for data storage compared to tools like Excel. There's a higher risk of data loss or misprocessing. Secondly, experiments are often collaborative efforts that may require external assistance. Sharing raw data in formats like Excel is more accessible, as not everyone can read or interpret program code.

Utilizing Classes and Encapsulation (Advanced CS Concepts to be Expanded)

Encapsulating experimental variables within classes is a beneficial practice, offering greater clarity than using plain arrays. If a variable has sub-variables—such as uncertainty or fundamental variables (a term I've coined to represent base measurements like length in calculating area)—it can be encapsulated within a class, such as an independent variable or a set of experiments.

Scalability

All the aforementioned practices serve to enhance scalability. Research and experiments are dynamic, evolving processes. The supporting programs should, as much as possible, build upon previous work while paving the way for future developments by retaining meaningful modules. This approach not only saves time but also serves as a valuable engineering log, showcasing progress and sparking new ideas.

Moreover, a program's scalability extends beyond individual experiments; it can serve as a template for future projects.
