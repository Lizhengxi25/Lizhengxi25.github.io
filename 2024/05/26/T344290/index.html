Cyclic Sequence

This problem comes from my school's problem set. I first attempted it in June 2023, and every couple of months when revisiting old problems, I would look at it again but still couldn't solve it. Recently, I decided to redo the previous problem set, especially focusing on the problems I hadn't solved before. I'm delighted to see that I can now tackle many problems that previously stumped me with greater ease. Last night, I finally figured out a solution to this problem, wrote it up today, and am recording it here.

Problem Description

Given a circular integer sequence of length ( n ): ( a_1, a_2, , a_n ). For any element ( a_i ) (where ( 1 i < n )), its next element is ( a_{i+1} ); specifically, the next element of ( a_n ) is ( a_1 ).

We want to cut this circular sequence to form a linear sequence of length ( n ) with a definite start and end. First, select ( a_k ) as the first element of the sequence, then sequentially append ( a_{k+1}, a_{k+2}, , a_n ) to the end, followed by ( a_1, a_2, , a_{k-1} ) in order.

Obviously, choosing different elements as the first element will yield different linear sequences of length ( n ).

Please calculate, among these ( n ) sequences, how many have non-negative prefix sums throughout.

Input Format

The first line contains an integer ( n ), representing the length of the initial circular sequence.

The second line contains ( n ) integers ( a_1, a_2, , a_n ), describing the circular sequence.

Output Format

An integer indicating the number of linear sequences that satisfy the condition.

Explanation

There are a total of ( 4 ) possible linear sequences:

  1. (-3, 5, 1, 2) (prefix sums: (-3, 2, 3, 5))
  2. (5, 1, 2, -3) (prefix sums: (5, 6, 8, 5))
  3. (1, 2, -3, 5) (prefix sums: (1, 3, 0, 5))
  4. (2, -3, 5, 1) (prefix sums: (2, -1, 4, 5))

Among them, the 2nd and 3rd sequences satisfy the requirement.

Data Constraints

  • For (25%) of the data, ( n ^3 ).
  • For (75%) of the data, ( n ^4 ).
  • For (100%) of the data, (1 n ^6), and (|a_i| ^3).

Solution

Since the data size can be up to (10^6), we need an algorithm with a time complexity of (O(n)) or (O(nn)). A brute-force method would have a time complexity of (O(n^2)): simulating each linear sequence and checking whether its prefix sums are non-negative. This approach isn't efficient enough, as checking all sequences would be too time-consuming.

If we can't efficiently solve the problem directly, perhaps we can think from a different angle. Initially, I considered that if all numbers in the sequence are non-negative, then every linear sequence would meet the requirement without any need for checks. However, if there are negative numbers, they will affect adjacent elements. So, perhaps we can focus on each negative number in the sequence.

Continuing with this idea, for a negative number itself, we can be certain that any sequence starting with it will be "invalid" because its first prefix sum is negative. But how does this negative number affect other elements to its left? By moving left from the negative number and accumulating the sum, if at any point the accumulated sum becomes negative, then sequences starting with these elements will also have negative prefix sums and thus be invalid.

Specifically, for a negative number ( a_i ), we can affect elements ( a_j ) where ( j i ), such that the accumulated sum ( s = a_j + a_{j+1} + + a_i ) is negative. All sequences starting at any ( a_j ) within this range are invalid. If the accumulated sum becomes non-negative, the influence of ( a_i ) stops.

It's important to note that when moving left, if we encounter another negative number, it becomes part of the accumulated sum. This negative number doesn't interrupt the influence but extends it further left.

Influencing elements to the right isn't helpful in this context because we're only concerned with sequences starting at different positions, and any influence to the right doesn't affect new starting positions.

Detailed Approach

We can implement the above idea using the following steps:

  1. Duplicate the Sequence: Since the sequence is circular, we can duplicate it to linearize the circularity. This allows us to handle wrap-around cases naturally.

  2. Traverse Backwards: Starting from the end of the duplicated sequence, we look for negative numbers.

  3. Accumulate Sums Leftward: For each negative number found, we move left, accumulating the sum. If the sum remains negative, we mark the starting positions as invalid.

  4. Skip Processed Elements: To optimize, we skip elements that have already been processed.
